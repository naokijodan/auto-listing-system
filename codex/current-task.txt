RAKUDAプロジェクト（~/Desktop/rakuda/）の出品パイプライン結合テストスクリプトを作成してください。

## 目的
テスト商品を作成し、画像→翻訳→ItemSpecifics→品質チェック→eBay出品の一連フローを検証するスクリプト

## ファイル
`apps/worker/src/__tests__/integration/listing-pipeline.test.ts` に作成

## テスト対象フロー
1. Product作成（テストデータ）
2. EnrichmentTask作成（翻訳・属性抽出）
3. 画像処理（processImagesForListing）
4. 品質チェックゲート（listing-quality-gate.ts の checkListingQuality）
5. eBay出品（publishToEbay - モック使用）

## 重要な参照ファイル
- `apps/worker/src/lib/ebay-publish-service.ts` - EbayPublishService クラス（createEbayListing, processImagesForListing, publishToEbay）
- `apps/worker/src/lib/listing-quality-gate.ts` - checkListingQuality関数
  - hardBlocks: 画像0枚、タイトル空or[EN]始まり、価格<=0
  - softWarnings: 説明空or[EN]始まり、ItemSpecifics充足率<80%
- `apps/worker/src/jobs/ebay-publish-worker.ts` - BullMQジョブ処理
- `packages/enrichment/src/translator.ts` - enrichProduct（翻訳）
- `packages/enrichment/src/item-specifics/extractor.ts` - extractItemSpecifics
- `packages/enrichment/src/item-specifics/field-definitions.ts` - getFieldsForCategory, resolveCategory

## テストケース

### 1. 正常フロー（Happy Path）
- テスト商品（時計カテゴリ）を作成
- EnrichmentTaskで翻訳・属性抽出を実行（OpenAI APIはモック）
- 品質チェックPASS → 出品成功（eBay APIはモック）

### 2. 画像なしでブロック
- 画像0枚の商品 → 品質ゲートでhardBlock → 出品中止

### 3. 翻訳失敗フォールバック
- OpenAI API失敗 → [EN]プレフィックス付きフォールバック → 品質ゲートでhardBlock

### 4. ItemSpecifics抽出テスト
- 時計商品の日本語タイトルからBrand, Model, Movement等を抽出
- DBのBrandテーブル（505件）とItemSpecificsFieldテーブル（109件）を使用

### 5. カテゴリ別プロンプト選択テスト
- Watchesカテゴリ → 「時計専用V2」プロンプトが選択される
- 未知カテゴリ → 「一般・汎用」デフォルトプロンプトが選択される

## モック方針
- **モック**: OpenAI API, eBay API（Inventory/Offer/Publish）, S3/MinIO画像アップロード
- **実DB使用**: Prisma（ローカルPostgres）- Brand, ItemSpecificsField, TranslationPromptは実データ使用
- **テストフレームワーク**: Vitest

## 既存テストの参考
- `apps/worker/src/__tests__/` 配下の既存テストを参考にスタイルを合わせる
- `vi.mock` でモジュールモック
- テスト後のDBクリーンアップを実装

## 注意
- eBay本番APIには絶対にリクエストを送らない
- テストデータは明確にテスト用と分かる名前にする（例: "TEST-INTEGRATION-..."）
- 各テストは独立して実行できること
